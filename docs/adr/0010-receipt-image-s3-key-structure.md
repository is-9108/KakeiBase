# ADR-0010: レシート画像のS3オブジェクトキーは年/月フォルダ階層+タイムスタンプファイル名とする

## ステータス

承認済み

## コンテキスト

ADR-0005でレシート画像バケットへのPresigned PUT URLによる直接アップロード方式を採用したが、S3オブジェクトキー(フォルダ構造・ファイル名)の命名規則は未定義だった。

ADR-0009が想定する利用シーン(月次の家計振り返り、確定申告での前年分領収書の確認)を踏まえると、S3コンソールやCLIから対象期間の画像を探す場面が発生する。フラットな1階層(全画像を同一プレフィックス直下に格納)では、件数が増えたときに月次・年次単位の絞り込みがしづらい。

想定利用シーンはいずれも「月」または「年」単位での参照であり、日単位で絞り込みたい場面は想定されていない。また想定件数(月数十件程度)では日フォルダを作っても中身は数件になり、日付情報自体はファイル名(`yyyyMMdd_hhmmss.fff.jpg`)からも読み取れるため、日フォルダを設けなくても検索性を大きく損なわない。

## 決定

**S3オブジェクトキーを `{yyyy}/{MM}/{yyyyMMdd}_{HHmmss}.{fff}.jpg` の形式にする。**

- 例: `2026/07/20260716_143052.418.jpg`
- 年→月の2階層をプレフィックスとし、月はゼロ埋め2桁。日フォルダは設けない(日付情報はファイル名に含まれるため)
- ファイル名の時分秒はゼロ埋め2桁、続く `{fff}` は実際のミリ秒値をゼロ埋め3桁で埋め込む(固定リテラルではない)
- アップロード可能な画像形式はJPEGのみに限定する(スマートフォンのカメラ撮影が主要な利用シーンのため)。拡張子はクライアントからの指定を使わず、常に `.jpg` 固定とする
- 階層・ファイル名の基準時刻は、Presigned URL発行時点(フロントエンドがS3へPUTする直前)のバックエンド側サーバー時刻(JST)とする
  - OCRで抽出される取引日はアップロード後にしか判明しないため、S3内の分類基準には使わない
  - ECS Fargateのコンテナ既定タイムゾーンはUTCであるため、OS/コンテナ設定に依存せず、アプリケーションコード内で `DateTime.UtcNow` から明示的にJSTへ変換する(`TimeZoneInfo` を使用し、環境差によるずれを防ぐ)
- キー自体はPresigned URL発行API(`POST /api/receipts/presigned-url`、ADR-0005参照)実装時にバックエンドが生成し、フロントエンドはそれをそのままアップロード先として使う(クライアント側でタイムスタンプを生成しない)

## 検討した代替案

| 案 | メリット | デメリット | 不採用の理由 |
|---|---|---|---|
| 年/月2階層 + ミリ秒タイムスタンプファイル名(採用) | 月次・年次の絞り込みが容易。ミリ秒により同一秒内の衝突を実質回避できる。想定利用シーンに対して階層がシンプル | 特定の日だけで絞り込みたい場合はファイル名を見る必要がある | ADR-0009の想定利用シーン(月次振り返り・確定申告)はいずれも月/年単位であり、日フォルダを設けても実利用に対して過剰なため |
| フラット構成 + UUIDファイル名(全ファイルを1プレフィックス直下に格納) | 実装が単純。衝突が原理的に発生しない | 件数が増えると月次・年次の絞り込みにS3のprefix検索を使えない | 検索性を優先する要件に反するため不採用 |
| 年/月/日の3階層まで設ける | 日単位でも絞り込める | 想定件数(月数十件)では日フォルダの中身が数件程度になり、階層を増やす割に実利用上の恩恵が薄い。日付情報はファイル名からも読み取れるため冗長 | 想定利用シーンに対してオーバースペックなため不採用 |
| ファイル名の秒未満部分を固定リテラル`.000`にする | 実装がわずかに単純 | 同一秒内に複数アップロードされた場合キーが衝突し、後勝ちで上書きされる | まとめて複数レシートを撮影・アップロードする利用シーンを否定できないため不採用 |
| クライアント(フロントエンド)側でタイムスタンプ生成 | バックエンド側の生成ロジックが不要 | クライアント時計のズレでフォルダ分類がずれる可能性がある | サーバー側の信頼できる時刻を基準にすべきため不採用 |
| アップロード元ファイルの拡張子をそのまま付与(jpg/jpeg/pngなど可変) | 元の形式が分かりやすい | クライアントが申告した拡張子と実際のバイト列が一致する保証がない(拡張子偽装の余地)。対応形式が増えるほどバリデーションが複雑になる | JPEGのみに用途を絞る方針(採用案)の方がシンプルかつ安全なため不採用 |

## 結果・影響

- `transactions.receipt_s3_key` カラム(ADR-0005、nullable text)への変更は不要。上記フォーマットのフルキー文字列(例: `2026/07/20260716_143052.418.jpg`)をそのまま格納する
- Presigned URL発行API実装時、バックエンドはサーバー側の現在時刻(UTC)をJSTへ明示変換した上でキーを生成する。このキー生成ロジックの単体テスト(境界値: 年跨ぎ、月跨ぎ、UTC→JST変換による日付繰り上がり、ミリ秒のゼロ埋めなど)を実装時に追加する
- アップロード可能な画像形式はJPEGのみとし、それ以外の形式はPresigned URL発行APIのFluentValidationで拒否する(Content-Typeベースの検証)。実際にPUTされたバイト列がJPEGとして不正だった場合は、OCR Lambda側のパース失敗時のエラー処理(ADR-0005: ログ出力のみ)に委ねる。この場合キー自体は変わらず`.jpg`のままなので、本ADRの命名規則としては追加対応不要
- 同一ミリ秒に複数のPresigned URL発行リクエストが重なった場合、理論上はキーが衝突しうる(ミリ秒精度は衝突を「実質排除」するが「保証」はしない)。個人利用規模の想定アップロード頻度ではリスクは無視できるレベルと判断し、リクエストID等の追加の一意性担保は行わない
- `infra/` 実装時、レシートバケットのTerraformモジュールとしては特にキー構造そのものへの制約は不要(バケット側でprefixを強制する仕組みは設けない。バックエンドのキー生成ロジックのみが命名規則を担保する)
- 将来、JPEG以外の形式(PDF、スクリーンショットのPNGなど)への対応が必要になった場合や、月間アップロード件数が大幅に増えて衝突リスクの再評価が必要になった場合は本ADRを見直す
