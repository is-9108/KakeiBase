# ADR-0011: クライアントサイドルーティングに React Router を採用する

## ステータス

承認済み

## コンテキスト

issue #10 (ログイン画面実装) により、フロントエンドに初めて複数ルート(`/login`, `/`)の出し分けが必要になった。
`docs/screen-design.md` の画面一覧には最終的に `/login`, `/`, `/transactions`, `/categories`, `/subscriptions` の5ルートが計画されており、今後も画面数は増える見込みである。

これまで `frontend/` にルーティングライブラリは導入しておらず、`App.tsx` は単一の内容を表示するだけだった。ルーティングの実現方法を決める必要がある。

## 決定

**React Router (`react-router-dom`)** を採用し、`BrowserRouter` + `Routes`/`Route` でルーティングを行う。

## 検討した代替案

| 案 | メリット | デメリット | 不採用の理由 |
|---|---|---|---|
| React Router (採用) | Reactエコシステムの事実上の標準で情報・実績が豊富。ネストルート・`useNavigate`等の機能が今後の画面追加(収支一覧・カテゴリ管理等)にそのまま使える | 新規依存パッケージが増える | — |
| 自前の pathname ベース簡易ルーティング(`useState` + `window.history` 直接操作) | 依存パッケージが増えない、実装が単純 | 5画面分のネスト・ガード処理・ブラウザ履歴操作を自前で作り込む必要があり、画面数が増えるほど保守コストが増大する。車輪の再発明でありポートフォリオとしての設計力アピールにも繋がらない | 将来の拡張性・保守性の観点で不採用 |
| TanStack Router | 型安全性が高い、TanStack Queryとの親和性 | 本プロジェクトでの採用実績・情報量がReact Routerより少なく、学習コストに見合うメリットが薄い | 不採用 |

## 結果・影響

- `frontend/package.json` に `react-router-dom` を新規依存として追加する
- `main.tsx` で `App` を `BrowserRouter` でラップし、`App.tsx` に `Routes` を定義する
- 今後の画面追加(収支一覧・カテゴリ管理・サブスク管理)は同じ `Routes` 配下にルートを追加していく方針とする
- 認証が必要なルートのアクセス制御(`ProtectedRoute` 等)は本ADRのスコープ外とし、必要になった時点で別途設計する
