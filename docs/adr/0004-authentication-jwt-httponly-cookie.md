# ADR-0004: 認証はASP.NET Core自前JWT + httpOnly Cookie + リフレッシュトークンで実装する

## ステータス

承認済み

## コンテキスト

KakeiBaseは個人の家計簿アプリであり、ユーザーは自分1人のみ。ユーザー登録機能は不要で、
あらかじめ決めた資格情報でログインし、以降はJWTで認証を行う。

以下の制約・前提のもとで認証方式を選定した。

- シングルユーザー運用のため、IDプロバイダーの管理コストをかけたくない
- フロントエンド(React SPA)からAPIを呼び出す構成のため、クレデンシャルをブラウザに安全に保持する方法が必要
- ポートフォリオとして「セキュリティへの意識」を示したい

## 決定

**ASP.NET Core自前JWT発行 + httpOnly Cookie保存 + リフレッシュトークン** を採用する。

- `POST /api/auth/login` でID/パスワードを検証し、アクセストークン(15分)とリフレッシュトークン(7日)を発行する
- 両トークンはレスポンスのhttpOnly Cookie としてセットする(JavaScriptからアクセス不可)
- `POST /api/auth/refresh` でリフレッシュトークンを検証し、新しいアクセストークンを発行する
- `POST /api/auth/logout` でCookieをクリアし、リフレッシュトークンを無効化する
- パスワードはBCryptでハッシュ化してDBに保存する。シークレット自体は環境変数に置かず、マイグレーション時のシードデータとして1件のみ投入する
- ユーザーID(GUID)は固定値とし、`users` テーブルへのシードは初回マイグレーション時に実行する

## 検討した代替案

| 案 | メリット | デメリット | 不採用の理由 |
|---|---|---|---|
| ASP.NET Core自前JWT + httpOnly Cookie + リフレッシュトークン(採用) | XSS耐性あり、Cookieの自動送信でフロント実装がシンプル、セキュリティ意識をアピールできる | CSRF対策が必要、CORS設定が必要 | — |
| Amazon Cognito | マネージドで認証の実装が不要、MFAなど将来の拡張が容易 | シングルユーザーに対してオーバースペック、月額コスト・設定の複雑さに見合わない | ポートフォリオ規模に不釣り合いなため不採用 |
| localStorage保存のJWT | フロント実装が最もシンプル | XSS攻撃でトークン窃取リスクがある | セキュリティ上の問題が明確なため不採用 |
| リフレッシュトークンなし(長期アクセストークン) | 実装がシンプル | トークン漏洩時の影響期間が長い、ポートフォリオとして不十分 | セキュリティ・ポートフォリオ評価の観点で不採用 |
| Auth0 / Firebase Authentication | 実装コストが低い | 外部サービス依存、自前実装より設計力のアピールに繋がらない | 不採用 |

## 結果・影響

- **CSRF対策が必要**: Cookie自動送信のため、状態変更リクエスト(`POST`/`PUT`/`DELETE`)にはCSRFトークンまたは`SameSite=Strict`属性で対策する。本プロジェクトでは`SameSite=Strict`を採用し、クロスオリジンリクエストでのCookie自動送信を防ぐ
- **CORS設定が必要**: フロントエンドのオリジン(開発: `http://localhost:5173`、本番: CloudFrontドメイン)を明示的に許可する
- **リフレッシュトークンの保管**: DBに保存しハッシュ化して管理する。トークンローテーション(使用済みリフレッシュトークンの無効化)を実装する
- シングルユーザーのため、将来マルチユーザー化する場合は認証方式の全面見直しが必要になる
