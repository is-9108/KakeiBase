# ADR-0005: レシート画像の OCR 処理を Lambda + Textract に委譲する

## ステータス

提案

## コンテキスト

レシート画像を撮影・アップロードするだけで収支を自動登録したい。
画像のアップロード先と OCR 処理をどこで行うかを検討した。

## 決定

フロントエンドが API サーバー経由で取得した Presigned URL を使って S3 に直接アップロードし、
S3 イベント通知で Lambda を起動して Amazon Textract で OCR を実行、結果を `transactions` テーブルに自動 INSERT する。

## 検討した代替案

| 案 | メリット | デメリット | 不採用の理由 |
|---|---|---|---|
| 採用案: S3 → Lambda → Textract | ECS への画像処理負荷なし。スパイク耐性が高い。責務分離が明確 | Lambda を別途実装・デプロイ。インフラが増える | — |
| Web API (ECS) で Textract を呼ぶ | インフラがシンプル。アプリコードに閉じる | 画像処理のスパイク負荷が ECS コンテナに直撃する。大きな画像バイナリを API サーバー経由で転送するのは非効率 | ECS に不要な責務を持たせたくないため不採用 |
| 手動入力のみ | 実装が不要 | レシート OCR というコア機能が失われる | 機能価値がないため不採用 |

## 結果・影響

- `transactions` テーブルに `receipt_s3_key` (text, NULLABLE) カラムを追加する
  - 手動入力の取引は NULL のまま
  - OCR 経由の取引は S3 オブジェクトキーを保持し、後から画像を参照可能にする
- S3 バケット（レシート画像専用）を新設する
- API サーバーに `POST /api/receipts/presigned-url` エンドポイントを追加する
  - フロントエンドは取得した Presigned PUT URL へ直接アップロードする（API サーバーを経由しない）
- Lambda の実装言語は Python 3.12（Boto3 の Textract AnalyzeExpense API の事例が充実しているため）
- パース失敗時はログ出力のみとし、DLQ（デッドレターキュー）による再処理は将来対応とする
- 画像の保持期間は無期限ではなくADR-0009で定める（365日後に削除する）
- S3オブジェクトキーのフォルダ階層・ファイル名の命名規則はADR-0010で定める（年/月フォルダ+タイムスタンプファイル名）
- 詳細は [architecture.md](../architecture.md) の「6. サーバーレス処理 (Lambda)」セクションを参照
