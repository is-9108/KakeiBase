# ADR-0006: 月次レポートを Lambda + SES で配信する

## ステータス

提案

## コンテキスト

前月の家計サマリ（収入合計・支出合計・カテゴリ別内訳）をメールで受け取りたい。
バッチ処理の実行基盤とメール配信サービスの選定を検討した。

## 決定

EventBridge Scheduler で毎月1日 08:00 JST に Lambda を起動し、
前月の収支を集計した HTML メールを Amazon SES で送信する。

## 検討した代替案

| 案 | メリット | デメリット | 不採用の理由 |
|---|---|---|---|
| 採用案: EventBridge → Lambda → SES | サーバーレスで常時起動コストゼロ。AWS サービス内で完結 | Lambda を別途実装・デプロイ | — |
| ECS 上の BackgroundService / Hangfire | アプリコードに閉じる。インフラが増えない | ECS コンテナが常時起動している必要がある。バッチとアプリの責務が混在する | ECS を常時起動しているコストが増加する。ADR-0003 のサブスク Lambda と同様の設計を踏襲したいため不採用 |
| SendGrid 等 AWS 外部のメール配信 | 配信実績・到達率が高い | AWS 外部依存が増える。追加コストが発生する | AWS サービスで統一できる SES で十分なため不採用 |
| PDF 添付メール | 印刷・保存に便利 | PDF 生成ライブラリの導入が必要。複雑化する | HTML メールで十分な情報量が伝わるため不採用 |

## 結果・影響

- EventBridge Scheduler ルールを新設する（毎月1日 08:00 JST）
- ADR-0003 のサブスクスケジューリング Lambda と同じ Lambda + EventBridge パターンを踏襲する
- メール件名: `[KakeiBase] YYYY年M月 月次レポート`
- 本文は HTML メール（収入合計・支出合計・残高・カテゴリ別内訳）
- Lambda の実装言語は Python 3.12（ADR-0005 と統一し、運用コストを下げる）
- SES の送信元ドメインは本番環境のドメインで検証済みであることを前提とする
- 詳細は [architecture.md](../architecture.md) の「6. サーバーレス処理 (Lambda)」セクションを参照
