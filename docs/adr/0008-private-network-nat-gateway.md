# ADR-0008: ネットワークは完全プライベート構成とし、外向き経路はNAT Gateway + S3 Gateway型エンドポイントを採用する

## ステータス

承認済み

## コンテキスト

個人利用のアプリケーションであっても、本番運用を想定したネットワーク分離とアクセス制御を設計・実装できることをポートフォリオとして示したい。以下の要件を前提に、VPC設計・セキュリティグループ設計・プライベートサブネットからの外向き通信経路を検討した。

- RDS・ECSはプライベートサブネットに配置し、インターネットから直接到達できないようにする
- アプリケーションへのアクセスは自分のみに限定する(自宅IP、外出時は自宅へのVPN接続経由)
- S3・LambdaもIAM/バケットポリシーで最小権限に絞る
- デプロイ先リージョンは東京(ap-northeast-1)

プライベートサブネットに配置したECS/Lambdaは、以下の外部AWSサービスへの通信が必要になる。

- ECS: ECRからのイメージpull(`ecr.api` / `ecr.dkr`、イメージレイヤー本体はS3)、CloudWatch Logsへのログ送信
- OCR Lambda(ADR-0005): Textract、S3(レシート画像)
- 月次レポートLambda(ADR-0006): SES
- ECS Exec(DB管理アクセス用): SSM Messages

## 決定

### 1. サブネット構成

- **パブリックサブネット**: ALB、NAT Gateway のみ
- **プライベートサブネット(アプリ)**: ECS Fargate、Lambda×3(OCR / 月次レポート / サブスク生成)
- **プライベートサブネット(DB専用)**: RDS PostgreSQL

### 2. セキュリティグループはSG間参照で連鎖させる

CIDRでの許可はALBの外側(自宅IP)のみとし、内部はすべてSGからSGへの参照で許可する。

| SG | インバウンド | アウトバウンド |
|---|---|---|
| `alb-sg` | 443 を自宅IPのみから許可 | `ecs-sg` の 8080 のみ |
| `ecs-sg` | 8080 を `alb-sg` からのみ | `rds-sg` の 5432、外向き 443 |
| `lambda-sg` | なし(Lambdaはインバウンドを受けない) | `rds-sg` の 5432、外向き 443 |
| `rds-sg` | 5432 を `ecs-sg` と `lambda-sg` からのみ | なし |

### 3. アクセス制御は「自宅IPのみ」とする

- ALBのSGで自宅IP(/32)のみ許可。外出時は自宅へのVPN接続経由でアクセスする
- WAFやBasic認証による一般公開はしない。採用選考ではREADMEのスクリーンショットとデモ動画で見せる

### 4. 外向き経路はNAT Gateway 1本(シングルAZ) + S3 Gateway型エンドポイント

- NAT Gatewayをパブリックサブネットに1本配置する(シングルAZ)
- S3宛の通信のみ無料のGateway型VPCエンドポイントを経由させる(ECRイメージレイヤーのpullとレシート画像の取得がS3宛のため、NATのデータ処理量も削減できる)

### 5. S3はネットワークではなくポリシーで締める

- フロント配信バケット: Block Public Access 全ON + CloudFront OAC(Origin Access Control)からのみ許可
- レシートバケット: Block Public Access 全ON、書き込みはPresigned URL経由のみ(CORSでPUT元をCloudFrontドメインに限定)、読み取りはOCR LambdaのIAMロールのみ

### 6. RDSへの管理アクセスは踏み台を置かない

- EF Coreマイグレーションはデプロイパイプライン内でECSのone-offタスク(`ecs run-task`)として実行する
- アドホックなDB調査はECS Exec(SSM経由)で稼働中コンテナからpsqlを実行する
- 踏み台EC2は設置しない(管理対象と固定費の削減)

## 検討した代替案(外向き経路)

東京リージョンの料金試算(2026年7月時点)に基づく。

| 案 | 月額概算 | メリット | デメリット | 不採用の理由 |
|---|---|---|---|---|
| NAT GW 1本 + S3 Gateway型EP(採用) | 約$45(NAT時間課金$0.062/時。個人利用のためデータ処理$0.062/GBはほぼゼロ) | シンプル、必要なサービスが増えても追加費用なし | インターネットへの経路自体は存在する(SGとルーティングで制御) | — |
| Interface型VPCエンドポイントで統一 | 約$60〜70(1エンドポイント$0.014/時 ≒ 月$10 × 6〜7個: ecr.api / ecr.dkr / logs / Textract / SES / ssmmessages 等) | インターネット経路が物理的に存在しない、最も堅牢 | 必要サービス数が多く時間課金が積み上がる。低トラフィックではデータ処理料の削減額が固定費を大きく下回る | 本構成では「サービス種類が多く、データ量が少ない」ためNATより高くつき不採用 |
| NAT GWマルチAZ(2本) | 約$90 | AZ障害への耐性 | 個人利用に対して過剰な固定費 | 可用性要件に見合わないため不採用 |
| fck-nat(EC2 t4g.nanoによるNAT代替) | 約$4 | 圧倒的に安い | 自前管理のEC2が増える、コミュニティ製AMIへの依存、ポートフォリオとして「本番らしさ」が薄れる | マネージドサービスでの設計を見せる目的に合わないため不採用(選択肢として認知した上での判断) |
| ECS/Lambdaをパブリックサブネットに配置(NAT不要) | $0 | 最安 | 「RDS・アプリを直接インターネットに晒さない」という本ADRの前提要件に反する | 要件に反するため不採用 |

## 結果・影響

- 月額コストの全体像(東京): ECS Fargate 約$10 + ALB 約$20 + NAT GW 約$45 + RDS(t4g.micro) 約$15〜20 = **合計約$90〜95**。増分のほとんどはNAT GWであり、「完全プライベート構成のセキュリティ要件の値段」として説明できる
- 自宅IPは変動しうるため、SGの許可IPをTerraformの変数として管理し、変更時は `terraform apply` で更新する運用とする
- シングルAZのNAT GWは可用性の割り切りであり、AZ障害時は外向き通信が止まる。個人利用では許容し、要件が変わればマルチAZ化を再検討する
- 採用担当者が直接デモを触れないトレードオフを受け入れる。代替としてREADMEにスクリーンショット・デモ動画・本ADRを含む設計ドキュメントを充実させる
- Terraformの `network` モジュールにVPC/サブネット/SG/NAT/エンドポイントを集約し、SG間参照の連鎖をコードで表現する
- 構成図は [architecture.md](../architecture.md) を参照
