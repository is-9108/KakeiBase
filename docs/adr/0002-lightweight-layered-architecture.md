# ADR-0002: バックエンドは軽量レイヤードアーキテクチャを採用する

## ステータス

承認済み

## コンテキスト

当初、バックエンドはフルのクリーンアーキテクチャ(Domain / Application / Infrastructure / WebApi の4プロジェクト分離)を想定していた。
しかしKakeiBaseのドメインロジックは、収支登録・カテゴリ集計・予算進捗の計算程度であり、複雑度は高くない。DBの差し替え予定もなく、チーム開発でもない(個人ポートフォリオ)。

この状態で4プロジェクトに分離すると、以下の懸念がある。

- Repositoryインターフェースの実装が1つしかなく、抽象化のコストに対してリターンが小さい
- ボイラープレート(プロジェクト間参照、DI配線)が増え、テストやCI/CDに割ける時間が減る
- 「なぜこの構成にしたか」を「教科書通りだから」以外で説明しづらい

## 決定

バックエンドは **1プロジェクト内でのフォルダ分離による軽量レイヤードアーキテクチャ** を採用する。

- `KakeiBase.WebApi` 1プロジェクトの中に `Domain/` `Application/` `Infrastructure/` `Endpoints/` フォルダを切る
- 依存の方向は崩さない
  - `Domain` は何にも依存しない
  - `Application` は `Domain` のみに依存する
  - `Infrastructure` は `Application` のインターフェースを実装する形で依存する
- 依存方向のルールは `NetArchTest` を使ったアーキテクチャテストとしてCIに組み込み、レビューではなく自動で強制する
- ドメインロジックが実際に厚くなる箇所(予算進捗計算、定期支出の自動生成)は `Domain` 層にしっかりモデルとして書き、それ以外の単純なCRUDまで無理に層を分けない

## 検討した代替案

| 案 | メリット | デメリット | 不採用の理由 |
|---|---|---|---|
| 軽量レイヤード(採用) | ボイラープレートが少ない、依存方向のルールは維持できる、将来プロジェクト分割へ移行しやすい | 層の境界が「フォルダ」なので意識しないと崩れやすい | アーキテクチャテストで機械的に強制することでデメリットを緩和できるため採用 |
| フルクリーンアーキテクチャ(4プロジェクト分離) | 層の独立性が強制される、大規模化・チーム開発への耐性が高い | 現状の複雑度に対して過剰、DI配線などボイラープレートが多い | プロジェクトの規模に見合わないため不採用 |
| Vertical Slice Architecture(MediatR) | 機能ごとの独立性が高い、CRUD中心の機能と相性が良い | 「なぜこの層に置いたか」を面接で説明する材料としてはレイヤードより伝わりにくい | 初めての本格ポートフォリオとしては説明のしやすさを優先し不採用 |
| 素朴な3層(Controller→Service→Repository) | 最速で実装できる | ドメインロジックとインフラの境界が曖昧になりやすい、設計力のアピールに繋がりにくい | ポートフォリオの目的(設計力のアピール)に合わないため不採用 |

## 結果・影響

- プロジェクト構成がシンプルになり、実装速度とテスト・CI/CDへの時間配分を優先できる
- 依存方向のルールをテストで強制することで、「フォルダ分離だから緩い」という懸念を技術的に払拭する
- 将来ドメインロジックが複雑化した場合は、フォルダをそのままプロジェクトに昇格させる形でADR-0001の方針に沿って再度ADRを起票し、フルクリーンアーキテクチャへの移行を検討する
